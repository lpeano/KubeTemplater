# How KubeTemplater Works

KubeTemplater is designed to be simple and powerful. It automates the management of Kubernetes resources by using `ConfigMap`s as a source of truth.

This document explains the core reconciliation logic of the operator.

## The Reconciliation Loop

The operator continuously performs the following steps:

### 1. Watch for ConfigMaps

KubeTemplater watches all `ConfigMap` resources across all namespaces in the Kubernetes cluster.

### 2. Filter by Annotation

The operator filters these `ConfigMap`s and only processes those that have a specific annotation:

```yaml
metadata:
  annotations:
    kubetemplater.io/enabled: "true" # Note: In previous versions, this was kubetemplater.io: "true"
```

If a `ConfigMap` has this annotation, the operator will inspect its `data` section. If the annotation is removed or changed, the operator will no longer manage the resources previously generated by it.

### 3. Read `resources.yaml`

The operator looks for a key named `resources.yaml` within the `data` section of the annotated `ConfigMap`. The content of this key must be a YAML array, where each item represents a resource to be templated and applied.

A single `ConfigMap` can define multiple Kubernetes resources.

### 4. Render Go Templates

For each entry in the `resources.yaml` array, the operator performs a rendering step.

- It takes the Go template from the `template` field.
- It uses the key-value pairs from the `values` field as input for the template.

This allows you to create reusable templates and manage environment-specific configurations easily.

**Example Entry:**
```yaml
- name: my-nginx-deployment
  template: |
    apiVersion: apps/v1
    kind: Deployment
    metadata:
      name: {{ .deployName }}
    spec:
      replicas: {{ .replicas }}
      # ...
  values:
    - name: deployName
      value: "my-nginx-deployment"
    - name: replicas
      value: "3"
```
In this example, `{{ .deployName }}` becomes `"my-nginx-deployment"` and `{{ .replicas }}` becomes `"3"`.

### 5. Apply with Server-Side Apply (SSA)

After rendering the template into a valid Kubernetes manifest, the operator uses a **Server-Side Apply** patch to apply the resource to the cluster.

**Why Server-Side Apply?**
- **Declarative**: SSA allows the operator to declaratively state the desired end-state of the resource.
- **Ownership**: The operator becomes a "manager" of the fields it controls. Other controllers (or humans) can manage other fields of the same resource without causing conflicts.
- **Efficiency**: The apply request only sends the fields the operator cares about, and the Kubernetes API server handles the merging logic.

If the rendered resource does not exist, it will be created. If it already exists, it will be updated only if there are differences between the rendered manifest and the live state of the resource.

This entire process ensures that the resources in your cluster are always in sync with the templates and values defined in your `ConfigMap`s.
